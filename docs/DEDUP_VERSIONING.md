# 知识库去重与版本控制

## 功能概述

知识库现在支持智能去重和版本控制，避免重复归档相同内容，并支持课表、考试等动态内容的版本管理。

### 核心功能

1. **去重**：自动检测并跳过重复内容
2. **版本控制**：对于课表、考试等经常更新的内容，支持覆盖旧版本或保留历史
3. **智能判断**：根据内容相似度自动决定是覆盖还是保留

## 去重功能

### 去重原理

系统使用两种方式检测重复内容：

1. **内容哈希**：对清洗后的文本计算 MD5 哈希，完全相同的内容会被检测为重复
2. **文本相似度**：对于非版本控制类别，使用词汇重叠度计算相似度，相似度 > 95% 视为重复

### 去重配置

```env
# 启用去重（默认开启）
DEDUP_ENABLED=true
```

### 去重示例

**场景 1：完全相同的内容**

```
第一次归档："计算机2301班周一第1-2节语文"
第二次归档："计算机2301班周一第1-2节语文"  ← 跳过，完全相同
```

**场景 2：相似内容**

```
第一次归档："计算机2301班周一第1-2节语文，第3-4节数学"
第二次归档："计算机2301班周一第1-2节语文，第3-4节数学。"  ← 跳过，相似度 > 95%
```

**场景 3：不同内容**

```
第一次归档："计算机2301班周一第1-2节语文"
第二次归档："计算机2301班周二第1-2节数学"  ← 保留，不同内容
```

## 版本控制

### 版本控制对象

系统会对以下类别的内容进行版本控制：

| 类别 | 说明 | 示例 |
|------|------|------|
| `schedule` | 课表 | "计算机2301班周一课表" |
| `exam` | 考试 | "三年级数学期中考试安排" |

### 版本控制策略

系统支持三种版本控制策略：

#### 1. 覆盖模式（overwrite）

每次更新都删除旧版本，只保留最新版本。

```env
VERSION_STRATEGY=overwrite
```

**适用场景**：
- 只关心最新状态
- 存储空间有限
- 不需要历史追溯

**示例**：
```
版本1："计算机2301班周一第1-2节语文"
版本2："计算机2301班周一第1-2节数学"  ← 删除版本1，只保留版本2
```

#### 2. 保留模式（keep）

保留所有历史版本，不做任何删除。

```env
VERSION_STRATEGY=keep
```

**适用场景**：
- 需要完整的变更历史
- 需要追溯历史记录
- 存储空间充足

**示例**：
```
版本1："计算机2301班周一第1-2节语文"
版本2："计算机2301班周一第1-2节数学"  ← 保留版本1和版本2
```

#### 3. 智能模式（smart）- 推荐

根据内容相似度自动决定是覆盖还是保留。

```env
VERSION_STRATEGY=smart
```

**判断逻辑**：
- 相似度 > 80%：视为同一内容的更新，覆盖旧版本
- 相似度 ≤ 80%：视为不同内容，保留历史

**适用场景**：
- 课表微调（如调课）：覆盖旧版本
- 课表大改（如学期初和学期中）：保留历史

**示例 1：课表微调（覆盖）**
```
版本1："计算机2301班周一第1-2节语文，第3-4节数学"
版本2："计算机2301班周一第1-2节语文，第3-4节英语"  ← 相似度高，覆盖版本1
```

**示例 2：课表大改（保留）**
```
版本1："计算机2301班第一学期课表：周一语文，周二数学..."
版本2："计算机2301班第二学期课表：周一英语，周二物理..."  ← 相似度低，保留两个版本
```

### 版本控制配置

```env
# 启用版本控制（默认开启）
VERSION_CONTROL_ENABLED=true
# 版本控制策略：overwrite/keep/smart
VERSION_STRATEGY=smart
# 需要版本控制的类别（逗号分隔）
VERSION_CONTROLLED_CATEGORIES=schedule,exam
```

## 版本管理

### 查看版本历史

```python
from agent.knowledge_base_v2 import get_knowledge_base

kb = get_knowledge_base(school_dir, corp_id)

# 查看某个班级的课表版本历史
history = kb.get_version_history("计算机2301", "schedule")

for version in history:
    print(f"版本 {version['version']}: {version['timestamp']}")
    print(f"  内容: {version['text_preview']}")
    print(f"  是否最新: {version['is_latest']}")
    print(f"  来源: {version['sender_nick']}")
    print()
```

## 使用场景

### 场景 1：课表更新

**需求**：计算机2301班的课表进行了调整

**版本控制策略**：smart（智能模式）

**流程**：
1. 系统接收到新课表："计算机2301班周一第1-2节语文，第3-4节数学"
2. 提取实体键："计算机2301"
3. 查找同一实体的旧版本
4. 计算新旧内容相似度
5. 如果相似度 > 80%，覆盖旧版本；否则保留历史

**结果**：
- 如果只是调课（相似度高）：覆盖旧版本，只保留最新课表
- 如果是学期初和学期中课表（相似度低）：保留两个版本

### 场景 2：考试安排更新

**需求**：期中考试时间调整

**版本控制策略**：smart（智能模式）

**流程**：
1. 系统接收到新考试安排："三年级数学期中考试时间：11月20日"
2. 提取实体键："三年级数学期中考试"
3. 查找同一实体的旧版本
4. 计算新旧内容相似度
5. 如果相似度 > 80%，覆盖旧版本；否则保留历史

**结果**：
- 如果只是时间调整（相似度高）：覆盖旧版本，只保留最新安排
- 如果是考试内容大改（相似度低）：保留两个版本

### 场景 3：重复消息

**需求**：用户重复发送相同消息

**去重策略**：启用去重

**流程**：
1. 系统接收到消息："今天作业：练习册第15-16页"
2. 计算内容哈希
3. 查找是否存在相同哈希的内容
4. 如果存在，跳过归档；否则正常归档

**结果**：
- 完全相同的消息：跳过，避免重复
- 相似但不同的消息：正常归档

## 性能说明

### 去重性能

| 操作 | 耗时 | 说明 |
|------|------|------|
| 内容哈希计算 | <1ms | MD5 计算，非常快 |
| 哈希查找 | O(n) | 遍历所有分块，但内存操作很快 |
| 文本相似度计算 | O(n*m) | 对于非版本控制类别，可能较慢 |

### 版本控制性能

| 操作 | 耗时 | 说明 |
|------|------|------|
| 实体键提取 | <1ms | 正则表达式匹配 |
| 查找同一实体 | O(n) | 遍历所有分块 |
| 删除旧版本 | O(n) | 重建索引，较慢 |

### 优化建议

1. **限制版本历史数量**：可以设置最大版本数，避免无限增长
2. **定期清理旧版本**：使用 `clear_old_operation_logs` 清理过期数据
3. **选择合适的策略**：对于频繁更新的内容，使用 `smart` 或 `overwrite`

## 代码变更

### 新增字段

- `content_hash`: 内容哈希（用于去重）
- `version`: 版本号
- `is_latest`: 是否是最新版本
- `replaces_id`: 替换的旧版本 ID

### 新增方法

- `_find_duplicate()`: 查找重复内容
- `_find_chunks_by_entity()`: 查找同一实体的所有分块
- `_remove_chunks()`: 移除指定分块
- `_compute_text_similarity()`: 计算文本相似度
- `get_version_history()`: 获取版本历史

### 新增配置

- `DEDUP_ENABLED`: 是否启用去重
- `VERSION_CONTROL_ENABLED`: 是否启用版本控制
- `VERSION_STRATEGY`: 版本控制策略
- `VERSION_CONTROLLED_CATEGORIES`: 需要版本控制的类别
